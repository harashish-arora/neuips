Trees can do transfer learning — but not the same kind
The whole point of neural network transfer learning is that pretraining shapes the internal representations (the hidden layer weights). You pretrain on ΔG_solv, the neurons learn features like "this molecule is polar and interacts strongly with protic solvents," and those features transfer to solubility prediction. You literally copy the weights from one task to another.
Trees don't have transferable internal representations. A LightGBM model trained on ΔG_solv produces a set of decision trees — you can't "fine-tune" those trees on log S. The splits are fixed.
But there is a clean tree-compatible version of transfer learning: use the pretraining model's predictions as an additional feature. Train a ΔG_solv predictor on CombiSolv, generate ΔG_solv predictions for every SC³ row, and add that prediction as feature #321 alongside the 320 RDKit + temperature features. Then train LightGBM on SC³ as usual.
This is sometimes called "stacking" or "prediction-as-feature" transfer. It's simple, it's clean, and it directly tests whether ΔG_solv information helps LightGBM.

But I have real concerns about the scramble implementation
This result is extraordinary enough that it demands extraordinary verification. Before you put this in the paper, you need to be confident about several things:
1. Does the scramble actually preserve global descriptors? The agent says "RDKit global descriptors remain approximately valid" — but "approximately" is doing a lot of work. MolLogP, TPSA, and most RDKit descriptors are computed from the molecular graph, not just atom counts. If you rewire bonds randomly, LogP changes (it depends on fragments), TPSA changes (it depends on what's bonded to polar atoms), ring counts change, aromatic atom counts change. If the scramble destroys most RDKit descriptors, then LightGBM's 20% degradation is expected (it lost its features), and the experiment doesn't test what it claims to test.
